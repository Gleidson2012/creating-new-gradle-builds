Creating a new build using Gradle means that you become a build author. You still need to follow the same path as a build user, but in addition, you will want to understand the Gradle Build Language and its API.

A build user needs to be able to identify the project type and how such a project is structured. In your case, you know in advance what project type you want but you still need to understand the conventions and project structure. In addition, you also have to understand the basic declarative elements and underlying model of the project type you want to build.

What you’ll build

You will create a basic build script with skeleton code for building a Java application.

What you’ll need

  • About 20 minutes

  • A text editor or IDE

  • A Java Development Kit (JDK), version 1.7 or better

  • A Gradle distribution, version 3.3 or better

Create a project

One way to get your project started is via the init command, which is documented under the Build Init plugin chapter of the user guide. It will create a skeleton project with a Gradle build file based on the project type that you specify. For example, you can run the following to create a basic Java project:

$ gradle init --type=java-library (1)
1 Check out the documentation to find out what project types are currently supported.

To learn more about bootstrapping a Gradle build, particularly if you want or need to do it by hand, follow the quick start guide for the type of project you want:

  • Java, Groovy, Scala: you have a JVM project. The user guide also has more details about each of the language plugins if you want a bit more depth.

  • c, cpp, assembler, objective-c, objective-cpp: you have a native build.

  • android: you have an Android project

It’s not uncommon in multi-project builds to have subprojects of different types, particularly when you have a polyglot project or a separate project for documentation. For example, you may want to use one or more examples from the quick start guides. You’ll more information about multi-project builds under Next Steps.

Learn the key concepts

Gradle is a very powerful tool, which means you need a good understanding of the underlying model to create maintainable builds. Fortunately, there are just a few core concepts to understand:

  • Gradle’s build phases—read the first section of the build lifecycle chapter for a description of these.

  • The project API and properties

  • Tasks

  • File handling

  • Dependency management

Next steps

  • Building a Java application: Follow Getting Started Building Java Applications guide.

  • Learn about multi-project builds: If you don’t see a need right now to create a multi-project build, then you can safely skip this step—it’s easy to make the switch later on if necessary. But if you do want to split a project into multiple, dependent and independent modules, you will need to learn how to configure them and manage the inter-module relationships. That requires a much more in-depth understanding of the topic than was covered in the build user path. We recommend you read the entirety of the multi-project builds chapter of the user guide.

  • Learn Groovy: You can make a fair bit of progress with Gradle without learning the language that build scripts are written in, but ultimately you’ll limit yourself if you don’t learn the basics of Groovy. Head to the Groovy website to find out what learning resources are available or simply check out the Groovy fundamentals and Groovy Closures sections of the free Udacity course.
